# Trading Review Wiki 业务设计分析

> 聚焦交易方法与工作流，不涉及技术实现细节。
> 分析日期：2026-06-18

---

## 一、系统定位与核心理念

### 1.1 是什么

一个面向 **A 股短线交易者** 的 LLM 驱动个人知识库。不是行情软件、不是自动交易系统、不是选股器 —— 它是一个**让交易认知持续积累、可检索、可验证、可纠错的认知外脑**。

### 1.2 核心问题

系统设计围绕交易者终生面临的五个问题：

1. **我最近真正理解了什么？**（认知沉淀）
2. **哪些判断后来被验证，哪些被证伪？**（预测回溯）
3. **哪些错误反复出现？**（错误模式识别）
4. **哪些模式只在特定市场环境下有效？**（条件化知识）
5. **哪些结论有原始证据，哪些只是舆情或推测？**（证据强度分层）

### 1.3 核心原则

> **不以盈亏论对错，以是否符合系统论对错。**
> 系统内亏损也是对的；系统外盈利也是错的。

这决定了整个系统的评价框架：交易复盘的目的不是追悔盈亏，而是检验**每一笔交易是否在自身的规则框架内执行**。

## 二、三层知识架构（最核心的业务设计）

系统用严格的分层来区分**证据、知识、事实状态**：

### 2.1 Raw 原始资料层 —— 不可变证据

| 子目录 | 内容 | 写入纪律 |
|--------|------|---------|
| `raw/日复盘/` | 每日人工填写的复盘草稿 | 只追加，不改写，不删除 |
| `raw/交割单/` | 券商导出的 CSV/Excel 原始交割单 | 原始凭证，不可修改 |
| `raw/研报新闻/` | 券商研报、新闻、公告原文 | 保留来源口径 |
| `raw/聊天舆情/` | 微信群、讨论组的市场讨论 | 低强度证据，保留上下文 |
| `raw/截图/` | 盘面截图、分时图、关键信息截图 | 时间戳 + 描述命名 |

**业务逻辑**：Raw 层是「案发现场」，不能被后续认知篡改。这解决了交易者最大的认知偏差之一 —— 事后合理化（hindsight bias）。当你要验证「我当初为什么买这只股票」时，Raw 层保留的是买入那一刻的全部信息环境，而不是事后改编的故事。

### 2.2 Wiki 正式知识层 —— LLM 维护的结构化认知

| 页面类型 | 目录 | 业务含义 |
|---------|------|---------|
| **股票** | `wiki/股票/` | 个股档案：基本信息、当前理解、交易日志、消息面、技术要点、风险与证伪条件 |
| **策略** | `wiki/策略/` | 入场/出场条件、仓位管理、实战记录、胜率盈亏比、适用市场环境、迭代记录 |
| **模式** | `wiki/模式/` | 市场阶段识别、题材生命周期、资金行为模式、与相近模式的区别、历史案例 |
| **概念** | `wiki/概念/` | 产业链结构、方法论框架、关键事实与催化、核心受益标的、验证记录 |
| **错误** | `wiki/错误/` | 高频错误定义、心理根源、典型案例、如何避免、近期犯案记录 |
| **总结** | `wiki/总结/` | 日复盘、阶段总结、持仓追踪 |
| **人物** | `wiki/人物/` | 关键意见源、产业人物、交易风格影响 |
| **查询** | `wiki/查询/` | 高价值问答归档、深度研究结果 |
| **源文档** | `wiki/源文档/` | 需在 wiki 内引用的原文摘要或备份 |

每个 Wiki 页面必须包含 YAML frontmatter：

| 字段 | 业务含义 |
|------|---------|
| `confidence` | 高/中/低 —— 强制标注认知确定程度 |
| `status` | 活跃/归档/迭代中 —— 防止死页面污染检索 |
| `sources` | 指向 Raw 层原始证据 —— 可追溯 |
| `related` | 双链到其他 Wiki 页面 —— 构建知识网络 |
| `last_reviewed` | 最后审阅时间 —— 驱动新鲜度评分 |
| `tags` | 主题标签，不滥用为泛词 |
| `aliases` | 搜索别名、简称、历史叫法 |

### 2.3 Data 结构化数据层 —— 会变化的事实和记忆

| 子层 | 位置 | 业务含义 |
|------|------|---------|
| **时序事实账本** | `data/facts/temporal_edges.jsonl` | 跟踪订单、客户、价格、产能、政策、验证等会随时间变化的事实 |
| **长期记忆** | `data/brain/*.jsonl` | 纠错记录、历史预测、验证结果、用户偏好、交易卫语句、自训练事件 |

**这是该系统的关键创新** —— 把「会过期的判断」从「稳定知识」中分离出来。

## 三、时序事实层（Temporal Facts v1）—— 解决「判断会过期」的问题

### 3.1 为什么需要这一层

传统笔记/wiki 的致命问题是：**你无法区分「曾经被认为对的判断」和「现在仍然有效的判断」**。三个月前写的「XX公司拿到了苹果订单」，后来可能被澄清、被证伪、或者订单已交付完毕不再有催化价值。

Temporal Facts 用 **状态机** 管理每一条事实的生命周期：

| 状态 | 含义 | 在问答中的行为 |
|------|------|-------------|
| `active` | 当前仍可参考（含待验证事实） | 进入主证据 |
| `superseded` | 被后续事实替代 | 默认排除，除非显式查历史 |
| `invalidated` | 被证伪/撤回 | 默认排除 |
| `expired` | 时间窗口已过 | 默认排除 |

### 3.2 事实的「证据强度」分层

| 等级 | 来源 | 业务规则 |
|------|------|---------|
| **A** | 公告、财报、监管文件、可复现市场数据 | 可作为确认事实 |
| **B** | 券商研报、公司IR、可靠行业调研 | 可用，保留来源口径 |
| **C** | 专家会议、媒体报道、会议纪要 | 只能作为待验证线索，**不可写成确认结论** |
| **D** | 群聊、小作文、传闻 | 只能作为观察项，必须标注「未确认」 |

### 3.3 Predicate 词表 —— 28 种标准事实关系

业务设计上将交易中需要跟踪的关系标准化为 28 种 predicate，核心包括：

**催化类**：`HAS_CATALYST`（政策/订单/涨价/产品发布）、`HAS_ORDER`（订单）、`HAS_CUSTOMER`（客户关系）、`HAS_POLICY_SUPPORT`（政策支持）

**产能与技术类**：`HAS_CAPACITY`（产能/扩产）、`HAS_TECH_PROGRESS`（技术进展）、`HAS_SUPPLY_CONSTRAINT`（供给约束）

**验证类**：`PRICE_VALIDATED`（价格验证）、`VOLUME_VALIDATED`（放量验证）、`CUSTOMER_VALIDATED`（客户兑现）、`TECH_VALIDATED`（技术兑现）、`FUNDAMENTAL_VALIDATED`（财报兑现）

**风险类**：`HAS_RISK`、`HAS_CLARIFICATION_RISK`（澄清/否认）、`HAS_COMPETITION_RISK`（竞争）、`HAS_DEMAND_RISK`（需求不及预期）、`HAS_SUPPLY_CHAIN_RISK`（供应链卡点）

**关系类**：`VALIDATES`（后续来源验证旧事实）、`CONTRADICTS`（后续来源反驳旧事实）

### 3.4 原子事实粒度

一条 fact 只表达一个事实。不允许将「某公司拿到苹果订单、金额2亿、下半年放量、股价有望重估」写成一条 —— 必须拆成：客户关系 + 订单 + 验证窗口 三条独立 fact。

**业务意义**：当一个判断被证伪时，只需让对应的一条 fact 失效，而不污染同一家公司/概念的其他正确判断。

## 四、交易规则体系 —— 系统化的交易纪律

### 4.1 仓位规则

| 规则 | 阈值 | 业务逻辑 |
|------|------|---------|
| 单赛道最大仓位 | 25% | 防止单一方向过度暴露 |
| 单票最大仓位 | 10% | 个股风险控制 |
| 最低现金仓位 | 20% | 永远保留机动资金 |
| 分赛道限制 | 半导体25%/光伏20%/热管理15%/消费20% | 按行业特性差异化暴露 |

### 4.2 交易频率规则

| 规则 | 阈值 |
|------|------|
| 单日最大交易笔数 | 30 笔 |
| 单日最大买入笔数 | 15 笔 |
| 单日最大卖出笔数 | 10 笔 |
| 最大同时持仓数 | 20 只 |
| 卖出后冷静期 | 24 小时 |

### 4.3 情绪周期信号规则（最核心的择时框架）

系统定义了四个市场情绪阶段及对应行动：

| 信号 | 触发条件 | 行动 |
|------|---------|------|
| **冰点** | 上涨家数 < 1500 | 观望寻找买点，增加现金 |
| **复苏** | 涨停 > 50 且 炸板率 < 30% | 考虑买入 |
| **高潮** | 上涨 > 3500 或 涨停 > 150 | 减仓、止盈 |
| **退潮** | 跌停 > 80 或 炸板率 > 50% | 清仓、持币 |

配合**市场环境仓位策略**：

| 市场环境 | 仓位 | 策略 |
|---------|------|------|
| 增量日 | 20-30% | 日内新方向首板做回封 |
| 竞价增量日 | 10% | 日内新方向一字卡位 |
| 缩量日 | 0-10% | 只做低位新催化首板 |

### 4.4 风控规则

| 规则 | 阈值 |
|------|------|
| 硬止损 | -8% |
| 软止损 | -5% |
| 移动止盈 | +15% 回撤 |
| 固定止盈 | +20% |
| 单日最大亏损 | -3%（触及停止交易） |

### 4.5 五条禁止事项

1. **处置效应禁止**：卖出后 24 小时内禁止买回同一标的
2. **同日倒仓禁止**：禁止同一标的同日先卖后买
3. **追高禁止**：禁止在涨停价追买
4. **横跳禁止**：不要横跳，专注一个方向
5. **YY 禁止**：跟随市场，不主观臆断

### 4.6 四条交易铁律

1. 不要横跳，专注一个方向
2. 跟随市场，不主观 YY
3. **逻辑破坏即卖出**（而非盈亏触发卖出）
4. 买入前想清楚兑现什么

## 五、核心工作流 —— 预测-执行-验证-纠错闭环

### 5.1 总览

```
盘前预测 → 盘中执行 → 盘后复盘 → 摄入知识库 → 纠错沉淀 → 下一次预测更准确
```

### 5.2 每日复盘工作流（六部分结构）

**触发**：每个交易日收盘后（18:00 自动，或手动触发）

#### 第一部分：大盘环境评估

- 三个核心指数表现（上证、创业板、科创50）
- 情绪指标体系：涨跌家数（880005）、涨停/跌停/连板家数、炸板率
- **市场判断四问**：增量还是缩量？增幅多少？指数共振了吗？情绪周期在哪个阶段？
- **开盘判断回顾**：竞价量能对比、竞价强势方向、早盘判断是否与实际走势一致

#### 第二部分：主线与板块分析

- 今日最强板块 + 龙头个股
- 成交额排名靠前的板块（判断容量方向）
- 是否发生板块轮动？方向和节奏是否正常？
- 容量品种表现（成交额前10有无走强/突破）
- 赚钱效应评估（涨停溢价、昨日涨停今日表现、连板溢价）

#### 第三部分：自选股与持仓复盘

- 持仓个股逐一对照：持仓逻辑 vs 今日实际表现 → 是否仍符合预期 → 明日计划
- 关注个股复核心对表：关注原因 vs 今日表现 → 维持/调整结论
- 机会回顾：是否出现指数共振机会、拿货/出货分时、符合买点的机会

#### 第四部分：今日交易记录与对错分析（核心）

- 买入记录：股票、时间、价格、仓位、买入逻辑、买点类型
- 卖出记录：股票、时间、价格、盈亏、卖出逻辑、卖对了吗？
- **对错分析**（不以盈亏论对错）：
  - 做对的交易：对在哪里、原因分析、如何复用
  - 做错的交易：错在哪里、原因分析、改进措施（必须具体可执行）
- **系统执行检查五问**：符合选股标准吗？符合买点条件吗？仓位合适吗？逻辑清楚吗？能接受亏损吗？
- 盈亏统计：日收益、交易笔数、胜率、单笔最大盈/亏

#### 第五部分：明日计划

- 指数预判：增量/缩量？幅度？有无共振机会？
- 主线预判：今日主线可持续吗？有无新主线？轮动方向预判？
- 重点关注标的和预期买/卖点
- 仓位计划按市场环境匹配
- 风险提示：系统性风险？外部冲击？情绪高位？
- 交易铁律提醒

#### 第六部分：心态与反思

- 心态四问：情绪稳定吗？冲动交易了吗？后悔情绪？执行到位吗？
- 心态问题记录
- 改进提醒
- 今日最大感悟 + 明日特别注意

### 5.3 摄入工作流（从原始资料到知识沉淀）

```
prepare → api-run → finalize → 人工审阅 → apply --write
```

**关键业务约束**：

- `raw/**` 永远不写（保护原始证据）
- 新建页面前必须检查：index 目录、目标目录、aliases、tags、related —— 已有页面覆盖 70% 以上含义则合并更新
- 会变化的事实必须写入 `data/facts/` 而非 wiki 正文
- 摄入必须是 dry-run 先行，人工审阅后才正式写入

**摄入十步顺序**：

1. 读取原始资料，确认资料类型、日期、来源和可信度
2. 查找已有相关页面，优先更新，不急着新建
3. 将当日盘面数据写入 `总结/YYYY-MM-DD-日复盘.md`
4. 将个股交易记录写入 `股票/股票名.md`
5. 将策略迭代写入 `策略/策略名.md`
6. 将市场结构和题材生命周期写入 `模式/`
7. 将产业链、方法论和关键变量写入 `概念/`
8. 将错误和反思写入 `错误/`
9. 更新 `index.md`、`overview.md` 和 `wiki/logs/log-YYYY-MM-DD.md`
10. 对会变化、会失效、会被后续来源替代的事实，写入 `data/facts/temporal_edges.jsonl`

### 5.4 预测验证闭环

```
daily-loop premarket（生成盘前预测，写入 brain/predictions）
  → 交易日执行交易
    → daily-loop postclose（自动对照预测与市场实际表现，生成验证结果）
      → self-train（从预测-验证对中生成训练样本）
        → brain resolve（更新纠错记忆）
          → 下次 ask/daily-loop 自动纳入约束
```

**业务意义**：这是系统中最关键的「认知进化」机制。每次预测都会被记录，每次验证都会回溯预测，每次偏差都会生成纠错记忆。系统不是静态的知识库，而是在持续学习的认知系统。

### 5.5 周期维护工作流

| 周期 | 时间 | 工作内容 | 时长 |
|------|------|---------|------|
| **每日** | 收盘后 15 分钟内 | 完整六部分复盘 + 交易对错分析 | 15-30 分钟 |
| **每周** | 周日 20:00 | 模式总结、胜率统计、系统优化 | 30 分钟 |
| **每月** | 月末 21:00 | 月度总结、系统进化、下月计划 | 1 小时 |

**周度 Lint 检查项**：

| 检查项 | 要做什么 |
|--------|---------|
| 矛盾发现 | 找出页面间互相冲突的观点 |
| 模式提炼 | 从近期交易和资料里提炼重复出现的盈利/亏损场景 |
| 个股档案检查 | 找陈旧页面、重复股票页、不完整交易日志 |
| 策略有效性 | 更新胜率、适用环境和归档状态 |
| 孤儿页面 | 找没有入链或没有 index 入口的页面 |
| 概念合并 | 合并近义概念，拆分上下位关系 |
| 事实状态 | 检查 active facts 是否已过期、被替代或被证伪 |
| 进化史 | 记录关键顿悟和系统变化 |

## 六、多源信息检索（Ask）—— 交易研究的方法论

### 6.1 六源融合

当交易者提出一个问题时，系统同时从六个来源召回证据：

| 来源 | 编号 | 角色 | 业务规则 |
|------|------|------|---------|
| **Wiki 正式知识** | W1, W2... | 主知识层 | 概念、策略、模式优先 |
| **Raw 原始资料** | R1, R2... | 最新证据 | 近期研报、会议、舆情，按鲜度加权 |
| **图谱关系** | G1, G2... | 关联发现 | 从已知页面一跳/二跳扩展关联标的 |
| **时序事实** | F1, F2... | 结构化事实 | active 事实进主证据，失效事实默认排除 |
| **长期记忆** | M1, M2... | 纠错约束 | 只能做先验/偏好，不能单独证明市场事实 |
| **行情 SQL** | S1, S2... | 量价验证 | 只读辅助，不参与知识写入 |

### 6.2 六段式回答标准

每一个研究问题的回答必须包含：

1. **结论** — 直接回答
2. **证据链** — 每个结论对应哪个源的哪条证据（如 [W3][R1][F2]）
3. **分歧/反证** — 有没有相反的证据或观点？如果 brain memory 与当前证据冲突，必须写入此处
4. **后续验证** — 什么条件出现时这个结论需要修正？
5. **交易含义** — 对交易决策的实际影响
6. **引用来源** — 完整来源列表

### 6.3 源路由的智能分发

系统根据问题特征自动选择最佳数据源组合：

| 问题特征 | 优先数据源 |
|---------|-----------|
| 普通知识库问题 | wiki, graph, brain |
| 最近、研报、会议、催化、订单 | raw（鲜度优先） |
| 错误、模式、复盘、高开、接盘、纪律 | wiki, raw, graph, brain |
| 案例、事实、验证、预测、样本 | facts JSONL |
| 股价、收盘、涨跌幅、K线、近 N 日 | stock SQL（自动触发） |
| 关联、产业链、上下游、扩散 | graph（自动二跳） |

### 6.4 Ask 与 Ingest 的关键区别

| 维度 | Ask（研究问答） | Ingest（知识摄入） |
|------|---------------|------------------|
| 目标 | 回答一个问题 | 将资料转化为结构化知识 |
| Raw 策略 | 限量、鲜度优先、去噪 | 宽召回，避免遗漏候选 |
| 输出 | 六段式回答 | 暂存报告 → 人工审阅 → 写入 |
| 写入 | 只读 | 仅 apply --write 后写入 |

## 七、交易策略体系

系统支持但不止于以下策略框架：

| 策略 | 核心逻辑 | 适用环境 |
|------|---------|---------|
| **容量票趋势交易** | 跟随成交额靠前品种的趋势 | 增量市场 |
| **极值策略反弹** | 在情绪冰点捕捉超跌反弹 | 冰点信号触发后 |
| **情绪周期套利** | 在复苏-高潮阶段做多，退潮阶段空仓 | 全周期（择时是关键） |
| **首板回封** | 日内新方向首板做回封 | 增量日 |
| **一字卡位** | 竞价增量日卡位新方向 | 竞价增量日 |

每个策略页面需维护：策略定义 → 入场条件 → 出场条件 → 仓位管理 → 实战记录 → 胜率盈亏比 → 适用环境 → 迭代记录。

## 八、错误管理系统

### 8.1 错误页面的标准结构

```
错误定义 → 心理根源 → 典型案例 → 如何避免 → 近期犯案记录 → 关联页面
```

### 8.2 常见的交易错误模式（系统追踪的）

- 情绪化交易（盘中临时起意）
- 止损执行不严（没有条件单或撤销条件单）
- 横跳问题（频繁换方向，追涨杀跌）
- 追高接盘（涨停价追买或在退潮信号下买入）
- 处置效应（卖出后立即买回同一标的）
- YY 交易（主观臆断，不跟随市场信号）

### 8.3 系统进化检验（三个月维度）

三个月后系统应能清晰回答：

1. **我的核心盈利模式是什么？**（容量趋势/极值反弹/情绪周期/指数共振）
2. **我最常犯的 3 个错误是什么？如何防范？**
3. **我在什么市场环境下最应该空仓？**（外部冲击/情绪退潮/无主线/连续亏损后）

```
周一至周五 盘中：
  ├── 用户交易执行
  ├── 原始资料积累（截图、聊天记录、研报阅读）
  └── 临时记录（raw/日复盘/ 草稿）

周一至周五 18:00：
  ├── OpenClaw 自动生成六部分日复盘报告
  ├── 用户补充感悟 → 存入 raw/openclaw数据/{日期}/
  └── 交割单自动归档（18:30）

用户说「摄入」时：
  ├── prepare → api-run → finalize → 人工审阅 → apply --write
  ├── Raw → Wiki（原始资料提炼为结构化知识）
  └── 会变化的事实进入 data/facts/

研究时（随时）：
  ├── ask 六源融合问答
  ├── 公司深度研究（company-research）
  └── 市场行情验证（market-validate）

周日 20:00 周度 Lint：
  ├── 矛盾发现 + 模式提炼 + 策略有效性更新
  └── 事实状态批量更新

月末 21:00 月度大 Lint：
  ├── 月度总结 + 系统进化记录
  └── 下月计划
```

## 十、业务设计的核心洞察

### 10.1 三个角色分工

| 角色 | 负责 | 不负责 |
|------|------|--------|
| **人类** | 提供原始资料、提出关键问题、裁决高风险判断 | 整理归纳、发现矛盾 |
| **LLM/Agent** | 归纳资料、维护 wiki、发现矛盾、补充双链 | 最终裁决、原始资料生产 |
| **程序化层** | 维护索引、日志、事实账本、审计、写入边界 | 内容判断 |

### 10.2 「一条事实一个权威位置」原则

系统严格定义每条信息类型的唯一权威位置：

| 信息类型 | 权威位置 | 其他页面怎么写 |
|---------|---------|-------------|
| 当日盘面数据 | `总结/YYYY-MM-DD-日复盘.md` | 只链接，不复制整段数据 |
| 个股交易记录 | `股票/股票名.md` | 只引用该股票页 |
| 概念定义 | `概念/概念名.md` | 只引用，不复制 |
| 模式识别条件 | `模式/模式名.md` | 只引用，不复制 |
| 舆情原文 | `raw/聊天舆情/YYYY-MM-DD.md` | wiki 只写提炼后的结论 |
| 会变化的事实 | `data/facts/temporal_edges.jsonl` | wiki 只解释含义和影响 |

### 10.3 知识新鲜度管理

系统通过多机制保证「旧知识不会污染新判断」：

- Frontmatter 的 `updated`/`last_reviewed` 驱动自动新鲜度评分
- 概念/股票/总结类页面超 180/365 天自动降权
- 策略/模式/错误类页面不做重罚（稳定经验不应因时间被遗忘）
- 时序事实的 active/expired/superseded/invalidated 状态机
- Raw 的 ask 模式按日期倒序扫描，鲜度优先

### 10.4 只读/写入边界的严格分离

| 操作 | raw/ | wiki/ | facts/ | brain/ |
|------|------|------|--------|--------|
| ask | 否 | 否 | 否 | 否 |
| api-run | 否 | 暂存 | 暂存 | 否 |
| apply --write | 否 | **是** | **是** | 否 |
| brain remember | 否 | 否 | 否 | **是** |
| daily-loop | 否 | **是** | **是** | **是** |
| hygiene | 否 | **是** | **是** | 否 |
| temporal-facts audit | 否 | 否 | 暂存 | 否 |

**业务意义**：这个边界矩阵保证了「证据不可篡改、知识写入需审阅、纠错记忆独立维护」的安全三角。

## 十一、总结

这套系统本质上是将**短线交易者的认知工作流**结构化为一套可运行的知识管理系统。它的核心不是技术创新，而是**业务建模的完整性**—— 从原始证据的不可变性保护，到知识的条件化表达（confidence/status/evidence level），到会变化事实的状态机管理，到错误模式的系统追踪，到预测-验证-纠错的闭环进化机制。

它回答的是一个比「今天买什么」更根本的问题：**一个交易者如何持续变聪明？**
