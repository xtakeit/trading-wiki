# A股个人投研助理 vs Trading Review Wiki —— 竞品分析与优化方案

> 分析日期：2026-06-18

---

## 一、两套系统定位对比

| 维度 | **我们的系统**（A股个人投研助理） | **TRW**（Trading Review Wiki） |
|------|--------------------------------|-------------------------------|
| **定位** | A 股投研知识库 + AI 辅助生成 | 短线交易者认知外脑 |
| **目标用户** | A 股个人投资者（泛投研） | A 股短线交易者（职业/半职业） |
| **核心场景** | 阅读关注人观点 → 蒸馏 → 复盘 → 研究 | 盘中交易执行 → 盘后复盘 → 认知纠错 → 预测验证 |
| **AI 角色** | 生成器（自动化内容生产） | 维护者 + 审阅者（知识整理 + 矛盾发现） |
| **状态** | ✅ 已实现，可运行 | ❌ 设计文档，未实现 |

**结论**：两套系统**互补性极强**。我们的强项是 AI 内容生成 + RAG 检索 + 可运行的 UI，TRW 的强项是业务建模深度（交易认知闭环、知识分层、时效管理）。

## 二、架构对比

### 2.1 知识分层

| 层 | TRW | 我们的系统 | 差距 |
|----|-----|-----------|------|
| **原始证据层** | `raw/` 五类（复盘草稿、交割单、研报、聊天舆情、截图），写入纪律严格（只追加、不改写、不删除） | `data/raw/` 三类（posts/news/market），有 snapshots 机制 | ⚠️ 我们缺「不可变性」的业务约束和审计机制 |
| **结构化知识层** | `wiki/` 九类页面（股票/策略/模式/概念/错误/总结/人物/查询/源文档），均有 `confidence`/`status`/`last_reviewed` 字段 | `data/` 六类文档（viewpoint/review/theme/stock/note/qa），仅 viewpoint 有 `confidence` | ❌ 缺 `status`（活跃/归档）、`last_reviewed`、错误/策略/模式等知识类型 |
| **时序事实层** | `data/facts/` 状态机（active/superseded/invalidated/expired），28 种标准化 predicate，A/B/C/D 证据强度 | `data/facts/predictions.jsonl` 有状态机（pending/confirmed/falsified/insufficient），但无证据强度、无标准化 predicate | ⚠️ 有基础但深度不足 |
| **长期记忆层** | `data/brain/` 纠错记录、预测、验证结果、交易卫语句 | 无独立 brain 层，依赖 RAG 检索 + QA 历史 | ❌ 缺失独立记忆系统 |

### 2.2 知识图谱

| 能力 | TRW | 我们的系统 |
|------|-----|-----------|
| 双向链接 | `related` 字段，跨页面双链 | 仅通过 `themes`/`stocks`/`tags` 实现弱关联 |
| 图谱遍历 | 一跳/二跳扩展（Ask 六源融合中的 G 源） | 无 |
| 实体标准化 | aliases 统一简称/历史叫法 | 无 |
| 孤儿页面检测 | 周度 Lint 检查无入链页面 | 无 |

### 2.3 信息检索（Ask/RAG）

| 维度 | TRW | 我们的系统 |
|------|-----|-----------|
| **数据源** | 六源融合（Wiki/Raw/Graph/Facts/Brain/SQL） | 单一 RAG 管道（chunks.jsonl） |
| **源路由** | 问题特征 → 自动选源（股价→SQL、关联→Graph、时效→Raw 鲜度优先） | 无智能路由，统一走混合检索 |
| **回答标准** | 六段式：结论 → 证据链 → 分歧/反证 → 后续验证 → 交易含义 → 来源 | 自由格式 + 引用标注 |
| **检索算法** | 未详述（依赖 LLM + 多源融合） | 混合评分：vector × 0.7 + keyword × 0.2 + metadata × 0.1 |
| **新鲜度** | `last_reviewed` 驱动自动降权，超 180/365 天自动降权 | 无时效加权 |

### 2.4 工作流

| 工作流 | TRW | 我们的系统 |
|--------|-----|-----------|
| **每日复盘** | 六部分结构（大盘评估→板块分析→持仓复盘→交易对错分析→明日计划→心态反思），包含交易记录和对错分析 | 十段式（市场环境→情绪周期→资金流向→主线→个股→观点共识→历史对照→观察点→风险→备注），**不含交易记录和心态反思** |
| **知识摄入** | 十步顺序（prepare→api-run→finalize→审阅→apply），dry-run 先行 | 生成→用户编辑→保存（简单直接） |
| **预测验证** | 盘前预测 → 自动对照 → 生成训练样本 → 更新纠错记忆 | 仅 facts 的多窗口手动验证 |
| **周期维护** | 日/周/月三级，含 Lint 检查（矛盾发现、模式提炼、事实状态更新等 8 项） | 无 |
| **交易执行** | 完整的交易记录、仓位管理、风控规则、情绪周期信号 | ❌ 完全没有交易执行模块 |

## 三、优缺点详细分析

### 3.1 我们的系统 —— 优势

#### ✅ 优势 1：已实现且可运行
完整 Next.js 15 App Router 应用，dashboard、列表页、详情页、新建/编辑页、RAG 调试页均已实现，构建通过，29 个测试通过。

#### ✅ 优势 2：AI 生成流水线成熟
- 四个 AI 生成类型（观点蒸馏、每日复盘、主题研究、个股档案）均走统一模式：types → prompts → AI wrapper → markdown renderer → API route → save
- Zod 结构化输出校验
- SSE 流式输出，用户体验好

#### ✅ 优势 3：本地 RAG 系统
- 零外部依赖的 embedding（FNV-1a 哈希 + 中文 bigram）
- 混合检索评分
- 支持元数据过滤（类型、主题、股票、标签、日期范围）
- 增量更新（upsert/remove）

#### ✅ 优势 4：多轮问答 + 来源追溯
- 会话线程持久化为 Markdown 文件
- 来源信息以 JSON 块形式嵌入文件，前后端均可解析
- 支持继续对话（threadId）

#### ✅ 优势 5：前端组件化良好
- AppShell + PageHero + Workbench 模式统一
- CSS Grid 响应式布局
- 思考过程 sticky 展示

#### ✅ 优势 6：事实的多窗口验证
- 每个可验证断言自动创建 8 窗口验证计划（1/3/5/10/20/30/90/180 天）
- 多数投票自动判定状态

### 3.2 我们的系统 —— 劣势

#### ❌ 劣势 1：知识缺少「状态」和「时效」管理
- 没有 `status`（活跃/归档/迭代中）区分，旧知识会污染检索结果
- 没有 `last_reviewed`，无法判断知识是否需要重新审阅
- RAG 检索结果不按新鲜度加权，半年前的旧资料和昨天的新资料权重相同

#### ❌ 劣势 2：事实层设计不够深
- 没有证据强度分级（A/B/C/D），所有事实「平等」对待
- 没有标准化 predicate（事实关系类型），claim 是自由文本
- 没有「过期/被替代」状态，只有 pending/confirmed/falsified/insufficient
- 事实只在 viewpoint 保存时自动生成，**theme 和 stock 不产出事实**

#### ❌ 劣势 3：缺少知识之间的关联
- 文档之间只有 themes/stocks/tags 的弱关联
- 没有双向链接（「相关页面」字段）
- 没有知识图谱遍历能力
- 无法回答「这个概念还涉及哪些股票/主题」这类关联问题

#### ❌ 劣势 4：没有交易执行层
- 没有交易记录（买/卖/仓位/盈亏）
- 没有仓位管理和风控规则
- 没有情绪周期信号框架
- 复盘中缺少「今日交易对错分析」和「心态反思」

#### ❌ 劣势 5：缺少纠错和进化机制
- 没有错误模式追踪（交易错误分类 → 根因 → 案例 → 防止）
- 没有预测-验证-纠错闭环
- 没有系统进化记录（「三个月后我比现在强在哪」无法回答）
- QA 对话只是存档，不会反哺知识库

#### ❌ 劣势 6：缺少知识新鲜度管理
- 没有按文档类型的差异化过期策略
- 概念/股票类知识可能过时，但没有自动降权
- 没有周期性的「知识审计」（矛盾发现、孤儿页面、重复合并）

#### ❌ 劣势 7：RAG 检索缺少源智能
- 所有 chunk 一视同仁，不区分原始证据 vs 加工知识 vs 个人判断
- 没有「图表/市场数据走 SQL」的异构数据源能力
- QA 回答格式缺少「反证/分歧」部分

## 四、优化方案

### 总策略：「三层递进」

```
Phase 1（立即可做）→ 补基础短板，无破坏性变更
Phase 2（2-3 周） → 知识质量升级，增加智能路由
Phase 3（长期演进）→ 交易闭环，认知进化
```

---

### Phase 1：知识质量基础增强 🟢 低风险 · 高收益

#### 1.1 Frontmatter 扩展 —— 给每篇文档加上「认知元数据」

**改动文件**：`lib/types/document.ts`、各保存函数、列表/详情 UI

```typescript
// 新增字段
interface DocumentFrontmatter {
  // ... existing ...
  /** 认知确定程度：high | medium | low */
  confidence?: string;        // ✅ 已有，需推广到全部类型
  /** 页面状态：active | archived | draft | iterating */
  status?: string;            // 🆕
  /** 最后审阅日期 */
  last_reviewed?: string;     // 🆕
  /** 关联文档 ID 列表 */
  related?: string[];         // 🆕
  /** 证据来源强度（仅事实类）：A | B | C | D */
  evidence_level?: string;    // 🆕
}
```

**收益**：
- 列表页可按 status 过滤（排除 archived）
- 详情页显示「最后审阅：N 天前」提醒
- related 实现基础双链

#### 1.2 RAG 新鲜度加权

**改动文件**：`lib/rag/retrieve.ts`

```
freshnessBoost = documentAge < 30天 ? 1.0 :
                 documentAge < 90天 ? 0.9 :
                 documentAge < 180天 ? 0.7 : 0.5;

finalScore = vectorScore × 0.6 + keywordScore × 0.15 + metadataScore × 0.1 + freshnessScore × 0.15
```

**收益**：新资料自动优先，旧资料不退场但不喧宾夺主。

#### 1.3 事实层扩展

**改动文件**：`lib/types/fact.ts`、`lib/storage/fact-store.ts`、`lib/themes/markdown.ts`、`lib/stocks/markdown.ts`

1. **增加过期/替代状态**：state 枚举新增 `expired`、`superseded`
2. **增加证据强度**：`evidenceLevel: 'A' | 'B' | 'C' | 'D'`
3. **主题研究和个股档案也产出事实**：在 theme research 和 stock profile 的 AI prompt 中增加 `verifiable_claims` 字段，保存时自动创建 facts

**收益**：
- 事实有了「时效性」管理
- 主题/个股研究的关键判断进入验证管道
- 证据强度让 Q&A 回答更有层次

#### 1.4 新增「错误记录」文档类型

**改动文件**：新建 `lib/types/mistake.ts`、`data/mistakes/`、相关的 API 和页面

```
type: mistake
title: "追高接盘 - 2026-06-15 某股票"
error_type: 追高接盘 | 止损不严 | 横跳 | 情绪化交易 | YY交易 | 处置效应 | 其他
psychological_root: "FOMO 心理，怕错过行情"
context: "当日涨停家数 80+，市场高潮信号已出现"
lesson: "高潮期禁止追买，严格执行买入条件单"
related_trades: ["trade-xxx-1"]
stocks: ["300604"]
date: "2026-06-15"
status: active | resolved
```

**收益**：将 TRW 的错误管理系统引入，是认知进化最关键的数据基础。

### Phase 2：知识网络与智能检索 🟡 中等风险 · 中高收益

#### 2.1 双向链接 + 知识图谱

**实现思路**：
1. **构建阶段**：扫描所有文档，基于 stocks/themes/tags 共现构建关联图
2. **存储**：`data/graph/edges.jsonl`，每条边 = `{from: docId, to: docId, weight: number, relation: string}`
3. **查询**：`expandFrom(docId, hops=2)` 返回关联文档
4. **展示**：详情页底部「相关文档」section

**收益**：从「信息孤岛」变成「知识网络」，详情页一键跳转关联内容。

#### 2.2 Ask 六段式回答 + 反证

**改动文件**：`app/api/ai/ask/route.ts` 的 SYSTEM_PROMPT

```
每个回答必须包含六个段落：

1. **结论**：直接回答
2. **证据链**：每个结论对应的来源编号（如 [W3][R1][F2]）
3. **分歧与反证**：有没有相反的证据？知识库中有无矛盾？
4. **后续验证**：什么条件出现时需要修正这个结论？
5. **交易含义**：对投资决策的实际影响
6. **引用来源**：完整列表
```

**收益**：从「AI 聊天」升级为「投研报告」，强制展示反方观点。

#### 2.3 RAG 源路由

**改动文件**：`lib/rag/retrieve.ts`、新增 `lib/rag/source-router.ts`

按问题特征智能选择检索策略：

| 问题特征检测 | 策略 |
|-------------|------|
| 含「最新/最近/今天/昨天」 | raw 类型加权 2× |
| 含「关联/产业链/上下游/扩散」 | graph 一跳扩展 |
| 含「验证/证实/证伪/预测」 | facts JSONL 优先 |
| 含「错误/复盘/教训/纪律」 | 新增 mistake 类型优先 |
| 默认 | 标准混合检索 |

**收益**：不同问题走不同检索路径，准确率大幅提升。

#### 2.4 自动矛盾发现（周度 Lint）

**新增**：`scripts/lint-knowledge.ts`

```bash
npm run lint:knowledge   # 手动运行
# 或在 cron 中周度触发
```

检查项：
1. **矛盾发现**：同一股票/主题下，不同文档的 stance 是否冲突
2. **孤儿页面**：没有 related 引用也没有 index 入口的页面
3. **过期事实**：active 事实是否已过验证窗口但未更新
4. **陈旧页面**：last_reviewed 超过 180 天未更新的概念/股票页面

### Phase 3：交易闭环 🟡 中等风险 · 高收益（面向真实交易者）

#### 3.1 交易日志文档类型

**新增**：`data/trades/`、`lib/types/trade.ts`

```
type: trade_log
date: "2026-06-18"
stock_code: "300604"
stock_name: "长川科技"
direction: buy | sell
price: 45.20
quantity: 1000
position_pct: 8.5
reason: "半导体设备板块复苏信号 + 放量突破前高"
trade_type: 首板回封 | 趋势跟随 | 极值反弹 | 情绪套利 | 其他
entry_signal: "涨停 > 50 且炸板率 < 30%"
profit_loss_pct?: -3.2
wrong?: false
mistake_type?: ""
lessons?: ""
```

**收益**：交易数据是投研闭环的最重要输入。没有交易记录，复盘就是空中楼阁。

#### 3.2 情绪周期信号页面

**新增**：`/sentiment` 页面

展示 TRW 的四阶段框架：
- 冰点（上涨 < 1500）
- 复苏（涨停 > 50 且炸板率 < 30%）
- 高潮（上涨 > 3500 或涨停 > 150）
- 退潮（跌停 > 80 或炸板率 > 50%）

用户手动输入当日数据，系统自动判定当前阶段并给出仓位建议。

#### 3.3 预测-验证-纠错闭环

在现有 facts 基础上增强：
1. **自动验证提醒**：dashboard 中「今日到期验证窗口」的 facts 高亮
2. **验证后自动更新**：confirm/falsify fact 后，自动更新关联文档的 `last_reviewed`
3. **纠错记忆**：falsified 的 fact 自动生成一条 brain memory，下次 Q&A 时作为先验约束

#### 3.4 仓位管理和风控规则

**新增**：`/risk` 页面

- 用户配置仓位规则（单票上限、总仓位上限等）
- 录入持仓后自动计算风险敞口
- 风控规则检查（是否有持仓触及止损线）

```
                    高收益
                      │
        Phase 1.1     │     Phase 2.1
        Frontmatter   │     知识图谱
        扩展          │     Phase 2.2
        Phase 1.2     │     六段式回答
        RAG新鲜度     │
        Phase 1.4     │     Phase 3.1
        错误记录      │     交易日志
                      │
  ────────────────────────────────────
  低风险              │              高风险
                      │
        Phase 1.3     │     Phase 2.4
        事实层扩展    │     自动矛盾发现
                      │
        Phase 2.3     │     Phase 3.2
        RAG源路由     │     情绪周期
                      │
                      │     Phase 3.3
                      │     预测验证闭环
                      │
                    低收益
```

**建议实施顺序**：

1. **本周**：Phase 1.1（Frontmatter 扩展）+ Phase 1.2（RAG 新鲜度）→ 立刻提升检索质量
2. **下周**：Phase 1.3（事实层扩展）+ Phase 1.4（错误记录）→ 补齐知识质量基础
3. **第三周**：Phase 2.2（六段式回答）+ Phase 2.1（知识图谱）→ 提升问答体验
4. **第四周**：Phase 2.3（RAG 源路由）+ Phase 2.4（周度 Lint）→ 智能检索 + 知识审计
5. **第二个月起**：Phase 3 系列 → 交易闭环，取决于用户是否实际交易

## 六、TRW 中不适合采纳的设计

| 设计 | 原因 |
|------|------|
| 28 种标准化 predicate | 对个人用户过于重型，自由文本 claim + 标签即可覆盖 80% 场景 |
| 六源融合的 SQL 行情源 | 我们的系统定位不含实时行情，且成本高 |
| 十步摄入工作流（prepare→api-run→finalize→apply） | 我们的 AI 生成 + 人工编辑模式已足够，增加环节反而降低效率 |
| 人物页面（wiki/人物/） | 关注人已通过 viewpoint 的 author 字段管理，不需要独立页面 |
| 源文档备份（wiki/源文档/） | raw 层已覆盖 |

## 七、总结

TRW 的设计文档提供了一套**极其完整的交易认知管理框架**，其核心价值不在于具体的技术选型，而在于对「交易者如何持续变聪明」这个问题的系统性回答。

我们的系统已经搭建了坚实的基础设施（Markdown 存储、AI 生成管道、RAG 检索、多轮问答、事实验证），但**在业务建模深度上明显不足** —— 知识缺少状态和时效管理、缺少关联网络、缺少错误追踪、缺少交易闭环。

建议按 Phase 1 → Phase 2 → Phase 3 的节奏推进，每个 Phase 都是独立可交付的增量，不会破坏现有功能，且每一步都有明确的验收标准。
